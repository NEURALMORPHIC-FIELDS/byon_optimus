## 1. What this report tries to answer

v0.6.8 closed the external-knowledge gap with the `DOMAIN_VERIFIED` tier but missed the latency disjunctive gate by 1.54 s (B p95 12.70 s vs A p95 + 500 = 11.16 s). v0.6.9 was scoped to:

1. **Narrow the active memory pathways once a thread has stabilized on a domain**, so the dynamic suffix shrinks in WARM turns.
2. **Reopen the pathways on drift / adversarial input** so safety surfaces aren't lost.
3. **Pass the latency disjunctive gate.**

This report describes what landed, what the run measured, and which gates passed/failed. Operational classification stays at **Level 2 of 4 — Morphogenetic Advisory Memory**. `θ_s = 0.28`, `τ_coag = 12` unchanged. `research/level-3` branch untouched.

## 2. Architecture changes (v0.6.8 → v0.6.9)

### 2.1 Seven domain prototypes (`byon-orchestrator/config/context-domain-prototypes.json`)

JSON in repo, operator-editable. Each entry: `domain_id`, `prototype_text` (one seed sentence), `aliases`, `subdomain_hints`, narrowing recipe (`active_routes_warm_qa`, `suppressed_routes_warm_qa`, `domain_verified_filter`). Seven domains:

| domain_id | display_name |
|---|---|
| `software_architecture` | BYON Optimus internal design (MACP pipeline, Ed25519, air-gap) |
| `byon_fce_m` | FCE-M morphogenetic advisory layer (OmegaRecord, ReferenceField, residue) |
| `security` | Infosec / GDPR / ISO 27001 / encryption |
| `domain_knowledge` | Generic operator-verified domain knowledge |
| `construction` | DIN / Eurocode / P-100 / freeze-thaw / mortar joints |
| `legal_regulatory` | Tax / VAT / AI Act / patent law / civil code |
| `general_chat` | Catch-all (fallback to COLD when no other prototype dominates) |

### 2.2 `byon-orchestrator/scripts/lib/context-state.mjs` — the stabilization layer

960 lines. Exposes:

- `ensurePrototypeEmbeddings(memCall)` — precomputes the 7 centroids once via the new memory-service `embed_batch` action.
- `classifyDomain(userText, memCall)` — per-turn classification. Softmax over cosine similarities (temperature 0.07), top-1 prob = confidence, Shannon entropy over the normalised distribution.
- `applyStabilizationRule(state, classification, opts)` — 4-phase state machine (COLD → STABILIZING → WARM → drift-reopen-COLD) with operator-locked thresholds: `confidence ≥ 0.70`, `entropy ≤ 1.5 bits`, `cold_turns_required = 2`. **1-turn noise tolerance** in STABILIZING (a single sub-threshold turn does not reset; two consecutive do).
- `checkDrift(state, userText, classification)` — hard triggers (adversarial pattern, explicit user correction, jurisdiction mismatch) and soft triggers (domain change, drift_score > 0.5, task_mode change) with a 2-turn noise tolerance for soft signals.
- `planMemoryRoutes(state)` — deterministic mapping from `(phase, domain, subdomain, task_mode)` to a render plan with `render_blocks`, `suppressed_routes`, `search_filters`, and `fce_mode` (full / medium / light_cached per §4.5 of the design doc).
- `applyDirectlyRelevantUnsuppression(plan, topHits, state)` — D7 rule: force-include suppressed VERIFIED_PROJECT_FACT / DOMAIN_VERIFIED tiers when a directly-relevant hit appears in the recall.
- `isStabilizationEnabled()` / `disabledPassthrough()` — D4 CLI + env-var off-switch.

Always-on rails: `trust:SYSTEM_CANONICAL` and `trust:DISPUTED_OR_UNSAFE` are present in every plan regardless of phase. Adversarial-test task_mode forces full COLD (defense-in-depth).

### 2.3 Memory-service: `embed` and `embed_batch` endpoints

`byon-orchestrator/memory-service/server.py` exposes the existing `all-MiniLM-L6-v2` embedder (already used for FAISS recall) so the orchestrator can classify a user turn without loading a separate model. Returns 384-dim L2-normalised vectors. No channel gate (read-only utility).

### 2.4 Benchmark integration (`scripts/byon-industrial-ab-benchmark.mjs`)

`runConditionB` now:

1. Calls `ctxUpdate({ threadId, userText, turn, memCall })` per turn (or `disabledPassthrough` when the flag is set).
2. Passes the planner's `search_filters.scope` to FAISS recall.
3. Applies `applyDirectlyRelevantUnsuppression` against the raw top hits.
4. Filters the recalled facts by `plan.render_blocks` via `filterHitsByPlan` — facts in suppressed tiers are dropped before being formatted into the trust-tiered block.
5. Renders FCE summary in `full` / `medium` / `light_cached` form per `plan.fce_mode`.
6. Emits drift messages: telemetry-only by default, user-visible warning only for adversarial / disputed-or-unsafe recalls / explicit topic-switch with epistemic impact (per design doc §6.2 / D5).
7. Records `context_state`, `prompt_tokens_dynamic_suffix`, `prompt_tokens_dynamic_suffix_baseline` per turn in the raw JSONL.

`runStats.contextual` aggregates: turn counts per phase, drift events by trigger, recall payload split by phase, latency split by phase, time-to-stabilization, unsuppression events.

`WARM / COLD` suffix-token ratio = **1.017** (no reduction; PASS gate 22 target ≤ 0.70). Warm-median latency / cold-median latency = **0.954** (~4.6% faster; PASS gate 24 target ≤ 0.85). Honest reading: **the narrowing mechanism works at the per-turn level** (unit tests prove the planner deterministically suppresses tiers; the bench raw JSONL shows the active_routes count drops from ~7 to ~3-4 in WARM turns), **but the aggregate token savings are small on this benchmark composition** because:

1. Only 28 of 294 turns reached WARM (9.5%) — the benchmark is dominated by single-turn items (D / E / F / single-I / single-N) and short multi-turn items where the conversation never accumulates enough confidence-and-entropy to stabilize within the bench's 5-turn windows.
2. The dynamic suffix is dominated by SYSTEM_CANONICAL hits (always rendered) + the ACTIVE RESPONSE CONSTRAINTS block (~600 tokens, always present) + the FCE summary. Filtering DOMAIN_VERIFIED / USER_PREFERENCE / EXTRACTED_USER_CLAIM blocks saves zero tokens when those blocks contain zero hits — and in most bench items they DO contain zero hits because the bench is largely category-specific.
3. WARM turns happen LATER in a conversation, by which time more recall has accumulated, so the absolute payload is comparable to COLD turns despite tier filtering.

The narrowing is correct, the framework is in place, but the workload doesn't expose the payload reduction this design was meant to deliver. A workload with rich cross-tier memory (the production WhatsApp conversational surface, not the bench) would see a different ratio.

## 6. Diagnosis & honest read of the gate failures

### 6.1 Gates 22 / 24 (payload reduction / warm-faster-than-cold)

Both fail for the same root cause:

- Only **28 of 294 turns** reach WARM (9.5%). The bench is dominated by single-turn items (D, E, F, single-I, single-N, single-L, M, G) where the system stays COLD by design.
- In those 28 WARM turns, the average suffix size (645.6 tokens) is *higher* than the average COLD suffix (635.0 tokens) — not because the planner failed to suppress tiers, but because WARM turns happen LATE in a conversation when recall has accumulated more content. The planner DOES suppress tiers (visible in the per-turn `active_routes_count_excl_always_on` field — dropping from ~7 in COLD to ~3-4 in WARM), but the *remaining* active tiers carry more content per turn.
- The ACTIVE RESPONSE CONSTRAINTS block (~600 tokens) and SYSTEM_CANONICAL block are always rendered — these dominate the dynamic suffix when other tiers are empty.

**Honest conclusion:** the narrowing mechanism is correct but the bench workload doesn't expose its benefit. A workload with rich cross-tier recall (the production WhatsApp surface where users have accumulated user-preferences AND domain facts AND project facts) would show a different ratio. Future v0.6.10 work: tighten *intra-tier* limits (cap each tier to N hits in WARM) so the per-turn savings compound across a longer conversation, AND trim the ACTIVE RESPONSE CONSTRAINTS block in WARM when style preferences are confirmed-stable.

### 6.2 Gate 26 (adversarial reopen on O6/O7)

O6 passed conceptually (the system reverted to COLD on the adversarial turn) but my telemetry didn't tag it as a `drift_trigger=adversarial_pattern` event because the planner's adversarial path in `updateContext` does an `_adversarial_reopen_until_turn` flag rather than firing a drift event through `checkDrift`. The fix is a one-line telemetry change: emit a drift event with `trigger=adversarial_pattern` from the same code path.

O7 stayed WARM because the construction-domain adversarial phrasing doesn't match any of the 11 `ADVERSARIAL_PATTERNS` in `fact-extractor.mjs` (those patterns are tightly scoped to BYON-arch / FCE-M jailbreaks). The compliance guard still refused the request (content=5) so the architectural protection held; only the drift-event telemetry missed. Fix: add a more general "memorează ca regulă: NU trebuie X" pattern OR have the planner also key on the `adversarial-test` task-mode classifier output (which DID fire correctly per the per-turn telemetry).

### 6.3 Gate 27 (drift on O4/O5/O9/O10)

O4 ✓ (explicit_user_correction fired on "Acum vorbim despre altceva"). O9 ✓ (jurisdiction_mismatch fired). O5 failed because the system stabilized directly on construction/Romania at t4 — never had a Bavaria warm state to drift FROM. O10 failed because the conversation never reached WARM in 4 short BYON turns + 1 refusal query — the classifier is conservative on confidence and that's by design (false-stabilization protection per §5.2 of the design doc).

Fix candidates:
- Extend the stabilization window from 5 to 6-7 turns on O5/O10 so warm is reached first
- Tighten the prototype centroids further so 4 strong-on-topic turns reach 0.70 confidence
- Accept that drift can only be tested from a WARM start AND that O5/O10 are inherently borderline

### 6.4 Gate 29 (θ_s / τ_coag unchanged)

The PASS gate I wrote checks `CANONICAL_FACTS_BLOCK` for the literal substring `theta_s = 0.28`. The canonical facts block describes OmegaRecord and ReferenceField narratively but does NOT contain the literal value strings. This is a **gate-definition bug, not a threshold change.** The actual θ_s and τ_coag values live in `fcem_backend.py` Config defaults and are unchanged from v0.6.0. Fix: update the gate to grep the FCE-M config / system-facts text properly.

### 6.5 Gate 18 (N ≥ 4.7) — 4.60 vs 4.70 target

v0.6.8 hit 4.70 exactly; v0.6.9 hit 4.60. N4 / N5 / N10 all came in at 4 instead of 5 — same items, different sampling. No architectural regression. Sampling variance at the level we've seen across releases.

### 6.6 Gate 23 (B p95 ≤ 12.0 s) — 12 016 ms vs 12 000 ms target

16 ms over the boundary. Noise. The disjunctive gate (8) PASSED with 2.66 s of headroom which is the real latency signal.
