# MACP v1.1 — Multi-Agent Control Protocol

**Specification for BYON Optimus Multi-Agent Orchestration**

> **v0.6.4a addendum.** `EvidencePack` is extended with an optional `fce_context: FceContextMetadata` field carrying the FCE-M morphogenetic summary (omega/contested/aligned counts, hashed center identifiers, advisory and priority counts). The field is metadata-only by construction and is rejected by the Auditor if any text-content keys (`label`, `description`, `content`, `text`, `name`, `title`) are present. See [`../src/types/protocol.ts`](../src/types/protocol.ts) for the type definition and [../../docs/RESEARCH_PROGRESS_v0.6.md](../../docs/RESEARCH_PROGRESS_v0.6.md) for the operational semantics. Worker plans, Auditor approves, Executor air-gapped — these invariants are unchanged.

## Overview

MACP (Multi-Agent Control Protocol) defines the document formats, handoff mechanisms, and security requirements for orchestrating AI agents in a secure, auditable manner.

## Document Types

### 1. InboxMessage

First entry point into the system. Represents an incoming task request.

```typescript
interface InboxMessage {
  message_id: string;        // Unique identifier
  channel_id: string;        // Source channel (telegram, email, etc.)
  content: string;           // Task description
  timestamp: string;         // ISO 8601 timestamp
  priority: "low" | "normal" | "high" | "urgent";
  metadata?: {
    source?: string;
    attachments?: string[];
    thread_id?: string;
  };
}
```

### 2. EvidencePack

Generated by Worker after analyzing the task and codebase.

```typescript
interface EvidencePack {
  document_type: "EVIDENCE_PACK";
  document_version: "1.0";
  evidence_id: string;
  timestamp: string;
  based_on_message: string;  // Reference to InboxMessage

  task_classification: {
    type: TaskType;          // "coding" | "general" | "research"
    confidence: number;      // 0-1
    keywords: string[];
  };

  codebase_context: {
    files_analyzed: string[];
    patterns_detected: string[];
    dependencies: string[];
    test_framework?: string;
    build_tool?: string;
  };

  extracted_facts: ExtractedFact[];
  memory_context: MemoryContext;
  hash: string;              // SHA256 of content
}
```

### 3. PlanDraft

Action plan proposed by Worker, pending Auditor review.

```typescript
interface PlanDraft {
  document_type: "PLAN_DRAFT";
  document_version: "1.0";
  plan_id: string;
  timestamp: string;
  based_on_evidence: string;  // Reference to EvidencePack

  intent: string;             // Human-readable goal
  actions: Action[];
  risk_level: "low" | "medium" | "high";
  rollback_possible: boolean;
  estimated_iterations: number;
  memory_context: MemoryContext;
  hash: string;
}

interface Action {
  action_id: string;
  type: ActionType;
  target: string;
  parameters: Record<string, unknown>;
  estimated_risk: "low" | "medium" | "high";
  rollback_possible: boolean;
}

type ActionType =
  | "code_edit"
  | "file_create"
  | "file_delete"
  | "file_write"
  | "file_modify"
  | "test_run"
  | "lint_run"
  | "build_run"
  | "shell_exec";
```

### 4. ApprovalRequest

Generated by Auditor when user approval is required.

```typescript
interface ApprovalRequest {
  document_type: "APPROVAL_REQUEST";
  document_version: "1.0";
  request_id: string;
  timestamp: string;
  based_on_plan: string;     // Reference to PlanDraft

  summary: string;
  risk_level: "low" | "medium" | "high";
  actions_preview: ActionPreview[];
  security_checks: SecurityCheck[];
  requires_approval: boolean;
  expires_at: string;        // ISO 8601

  user_options: UserOption[];
  hash: string;
}
```

### 5. ExecutionOrder

Signed authorization for the Executor to perform actions.

```typescript
interface ExecutionOrder {
  document_type: "EXECUTION_ORDER";
  document_version: "1.0";
  order_id: string;
  timestamp: string;
  based_on_plan: string;     // Reference to approved PlanDraft

  approved_by: string;       // User or "auto_approval"
  approved_at: string;
  actions: Action[];
  constraints: ExecutionConstraints;
  rollback: RollbackConfig;

  hash: string;              // SHA256 of content (excluding signature)
  signature: string;         // Ed25519 signature (base64)
}

interface ExecutionConstraints {
  max_iterations: number;
  timeout_minutes: number;
  memory_limit_mb?: number;
  disk_limit_mb?: number;
}
```

### 6. JohnsonReceipt

Execution result document generated by Executor.

```typescript
interface JohnsonReceipt {
  document_type: "JOHNSON_RECEIPT";
  document_version: "1.0";
  receipt_id: string;
  timestamp: string;
  based_on_order: string;    // Reference to ExecutionOrder

  execution_summary: {
    status: "success" | "partial" | "failed" | "rejected";
    actions_total: number;
    actions_completed: number;
    actions_failed: number;
    iterations_used: number;
    duration_ms: number;
  };

  action_results: ActionResult[];
  errors: ExecutionError[];

  changes_made: {
    files_modified: string[];
    files_created: string[];
    files_deleted: string[];
  };

  verification: {
    tests_passing: boolean | null;
    lint_passing: boolean | null;
    build_passing: boolean | null;
  };

  hash: string;
}
```

## Handoff Protocol

### Directory Structure

```
handoff/
├── inbox/                    # InboxMessage files
│   └── {message_id}.json
├── worker_to_auditor/        # EvidencePack + PlanDraft pairs
│   ├── evidence_{timestamp}.json
│   └── plan_{timestamp}.json
├── auditor_to_user/          # ApprovalRequest files
│   └── approval_{timestamp}.json
├── auditor_to_executor/      # ExecutionOrder files
│   └── order_{timestamp}.json
└── executor_to_worker/       # JohnsonReceipt files
    └── receipt_{receipt_id}.json
```

### File Naming Convention

| Document Type | Pattern |
|---------------|---------|
| InboxMessage | `{message_id}.json` |
| EvidencePack | `evidence_{timestamp}.json` |
| PlanDraft | `plan_{timestamp}.json` |
| ApprovalRequest | `approval_{timestamp}.json` |
| ExecutionOrder | `order_{timestamp}.json` |
| JohnsonReceipt | `receipt_{receipt_id}.json` |

### Processing Order

1. **Worker polls** `inbox/` for new messages
2. **Worker writes** evidence + plan pair to `worker_to_auditor/`
3. **Auditor watches** `worker_to_auditor/` for plan pairs
4. **Auditor writes** approval request to `auditor_to_user/`
5. **User approves** via API endpoint
6. **Auditor signs** and writes order to `auditor_to_executor/`
7. **Executor watches** `auditor_to_executor/` for orders
8. **Executor writes** receipt to `executor_to_worker/`

## Security Requirements

### Ed25519 Signatures

All ExecutionOrders MUST be signed by the Auditor using Ed25519:

```typescript
// Signature creation (Auditor)
const orderWithoutSignature = { ...order };
delete orderWithoutSignature.signature;
const contentToSign = JSON.stringify(orderWithoutSignature, Object.keys(orderWithoutSignature).sort());
const signature = ed25519.sign(contentToSign, privateKey);
order.signature = base64.encode(signature);

// Signature verification (Executor)
const orderWithoutSignature = { ...order };
delete orderWithoutSignature.signature;
const contentToVerify = JSON.stringify(orderWithoutSignature, Object.keys(orderWithoutSignature).sort());
const isValid = ed25519.verify(contentToVerify, base64.decode(order.signature), publicKey);
```

### Hash Verification

All documents include a SHA256 hash for integrity verification:

```typescript
function calculateHash(document: Omit<Document, "hash">): string {
  const content = JSON.stringify(document, Object.keys(document).sort());
  return crypto.createHash("sha256").update(content).digest("hex");
}
```

### Air-Gapped Execution

The Executor agent MUST run with no network access:

```yaml
# docker-compose.yml
executor:
  network_mode: none
  read_only: true
  security_opt:
    - no-new-privileges:true
```

## Risk Classification

### Action Risk Levels

| Action Type | Default Risk | Notes |
|-------------|--------------|-------|
| file_create | Low | Creating new files |
| test_run | Low | Running tests |
| lint_run | Low | Running linter |
| build_run | Low | Running build |
| file_write | Medium | Writing to existing files |
| file_modify | Medium | Modifying content |
| code_edit | Medium | Editing source code |
| file_delete | High | Deleting files |
| shell_exec | High | Shell commands |

### High-Risk Targets

Files matching these patterns are automatically elevated to high risk:

- `.env*` - Environment variables
- `*secret*` - Secrets
- `*credential*` - Credentials
- `*password*` - Passwords
- `*.lock` - Lock files
- `config.prod*` - Production configs

## Error Handling

### Error Types

| Type | Recoverable | Action |
|------|-------------|--------|
| `validation_error` | Yes | Retry with correction |
| `signature_invalid` | No | Reject order |
| `action_failed` | Partial | Continue remaining |
| `timeout` | No | Generate partial receipt |
| `iteration_limit` | No | Generate partial receipt |
| `rejection` | No | Pre-execution rejection |

### Receipt Status Values

| Status | Meaning |
|--------|---------|
| `success` | All actions completed |
| `partial` | Some actions failed |
| `failed` | All actions failed |
| `rejected` | Pre-execution rejection (signature/validation) |

## Compliance

All implementations MUST:

1. Validate document hashes before processing
2. Verify Ed25519 signatures on ExecutionOrders
3. Log all document transitions to audit trail
4. Reject documents with missing required fields
5. Enforce risk-based approval requirements
6. Maintain tamper-evident audit chain
